# Phase 5.2: Long-Term Evolution & Community Governance Handoff

## Overview

This document defines the framework for progressive decentralization of the agentic tokenomics system, from core-team-operated infrastructure to fully community-governed operations. It covers mechanism parameter adjustment governance, agent persona evolution, community handoff milestones, open question resolution tracking, and a continuous improvement framework.

### Philosophy: Progressive Decentralization

The agentic tokenomics system is designed for **community ownership** from the outset, but achieves it through a deliberate, staged transition rather than immediate decentralization. This approach:

1. **Reduces launch risk** -- core team expertise guides early parameter tuning when real-world data is sparse.
2. **Builds institutional knowledge** -- the community learns governance patterns by progressively taking on more responsibility.
3. **Maintains accountability** -- at every stage, there is a clear responsible party for each operational domain.
4. **Ensures continuity** -- no single handoff event; instead, a gradient from centralized to decentralized across multiple dimensions.

> "The goal is not decentralization for its own sake, but a governance structure that faithfully serves the ecological mission while being resilient to any single point of failure." -- Tokenomics WG

The four governance layers (Layer 1: Fully Automated, Layer 2: Agentic + Oversight, Layer 3: Human-in-Loop, Layer 4: Constitutional) remain the foundational framework throughout the evolution. What changes over time is **who** fills each role and **how much** authority flows through automated versus human channels.

## Mechanism Parameter Adjustment Governance

Each mechanism deployed in the economic reboot has tunable parameters. This section specifies the governance process for adjusting each parameter post-deployment, including who can propose changes, what analysis is required, the voting process, and post-change monitoring obligations.

### General Principles

```yaml
parameter_change_principles:
  1_transparency:
    description: >
      All parameter change proposals must include a rationale, expected impact
      analysis, and rollback plan. No opaque changes.
  2_proportionality:
    description: >
      The governance effort required for a change is proportional to its
      economic impact. Minor operational tuning uses lighter processes than
      structural economic shifts.
  3_evidence_based:
    description: >
      Parameter changes should be supported by data — historical performance,
      simulation results, or peer network comparisons.
  4_monitoring_obligation:
    description: >
      Every parameter change triggers a mandatory monitoring period. The
      proposer (or core team during Stage 1) must publish a post-change
      assessment within the monitoring window.
  5_reversibility:
    description: >
      Proposals must include a rollback plan. If monitoring reveals adverse
      effects exceeding defined thresholds, the previous parameter value
      is restored via expedited governance.
```

### M013: Value-Based Fee Routing

#### Fee Rate Changes (Layer 3)

| Attribute | Specification |
|-----------|--------------|
| **Parameters** | `issuance_fee_rate`, `transfer_fee_rate`, `retirement_fee_rate`, `trade_fee_rate` |
| **Governance Layer** | Layer 3 (Human-in-Loop) |
| **Who Can Propose** | Any REGEN holder meeting deposit requirement; AGENT-003 may generate recommendations |
| **Deposit** | Standard governance deposit (currently 1,000 REGEN) |

```yaml
fee_rate_change_process:
  1_proposal_submission:
    required_analysis:
      - current_fee_revenue_breakdown: "Monthly revenue by transaction type for past 3 months"
      - projected_impact_model: >
          Simulation showing expected revenue change at proposed rate,
          using historical transaction volumes. Must include pessimistic
          scenario (volume drops 20%) and optimistic scenario (volume unchanged).
      - competitive_analysis: "Fee comparison with peer credit registries (Toucan, Gold Standard, Verra)"
      - user_impact_assessment: "Effect on median transaction cost for each user segment"
    optional_analysis:
      - agent_003_recommendation: "AGENT-003 Market Monitor fee optimization analysis"
      - blockscience_simulation: "cadCAD or equivalent simulation output"

  2_community_discussion:
    venue: forum.regen.network
    minimum_duration: 14 days
    required_artifacts:
      - forum_post_with_analysis
      - q_and_a_period
    agent_support:
      - AGENT-002 publishes impact summary and FAQ
      - AGENT-003 provides real-time scenario modeling on request

  3_on_chain_vote:
    voting_period: 7 days
    threshold: simple majority (>50%)
    quorum: standard governance quorum
    poa_variant: dual-track tally (validator 60% / holder 40%) when M014 active

  4_execution:
    delay: 24-hour execution delay after vote passes (safety window)
    action: ParameterChangeProposal updates x/feerouter config

  5_post_change_monitoring:
    monitoring_period: 90 days
    metrics_tracked:
      - daily_fee_revenue_by_type
      - transaction_volume_by_type
      - user_retention_rate
      - marketplace_listing_activity
    reporting: Monthly monitoring reports published to forum by proposer
    rollback_trigger: >
      If fee revenue drops >30% (relative to pre-change baseline, adjusted
      for market conditions) OR transaction volume drops >40%, emergency
      rollback proposal is filed with expedited 3-day voting period.
```

#### Distribution Share Changes (Layer 2)

| Attribute | Specification |

```yaml
distribution_share_change_process:
  description: >
    Distribution share changes are operational tuning within the agreed fee

    escalation: >
      If any share changes by more than 10 percentage points, escalate
      to Layer 3 (full community vote with impact analysis).

  3_monitoring:
    monitoring_period: 30 days
    metrics: pool_balance_trajectories, validator_compensation_adequacy
```

### M012: Fixed Cap Dynamic Supply

#### Hard Cap Changes (Layer 4 -- Constitutional)

| Attribute | Specification |
|-----------|--------------|
| **Parameter** | `hard_cap` |
| **Governance Layer** | Layer 4 (Constitutional) -- 67% supermajority required |
| **Who Can Propose** | Any REGEN holder; expected to be extremely rare |
| **Frequency Expectation** | Should essentially never change; comparable to a constitutional amendment |

```yaml
hard_cap_change_process:
  description: >
    The hard cap is the fundamental monetary policy of the REGEN token.
    Changing it is equivalent to a constitutional amendment. This should
    only be contemplated under extraordinary circumstances (e.g., the
    cap proves unworkable after years of operation).

  1_pre_proposal:
    required: >
      Formal RFC published to forum with minimum 30-day discussion period
      before on-chain proposal submission.
    required_analysis:
      - multi_year_supply_analysis: "Complete history of mint/burn dynamics"
      - economic_modeling: "Blockscience-grade simulation of new cap effects"
      - stakeholder_impact: "Effect on all participant classes"
      - precedent_analysis: "What other networks have changed fundamental supply parameters, and outcomes"

  2_on_chain_vote:
    voting_period: 14 days (extended)
    threshold: 67% supermajority
    quorum: elevated quorum (2x standard)
    poa_variant: Both validator track AND holder track must independently pass at 67%

  3_execution:
    delay: 7-day execution delay
    note: On-chain enforcement via Security Invariant — Cap Inviolability

  4_monitoring:
    monitoring_period: 12 months
    reporting: Quarterly supply health reports
```

## Agent Persona Evolution Process

Agent personas are not static. As the network matures and trust in autonomous systems grows, agents may gain new capabilities, have their autonomy levels adjusted, or be retired. This section defines the governance processes for each type of agent evolution.

### Adding a New Agent Persona

```yaml
new_agent_proposal:
  governance_layer: Layer 3 (Human-in-Loop)
  who_can_propose: Any community member (Stage 3), core team or validator body (Stage 1-2)

  proposal_template:
    required_fields:
      agent_id: "Unique identifier (AGENT-NNN)"
      name: "Human-readable name"
      type: "domain_specialist | infrastructure | cross_cutting"
      purpose: >
        Clear statement of what problem this agent solves and why existing
        agents cannot address it.
      domain: "Primary knowledge domain(s)"
      mechanisms_served: "Which mechanisms (M0XX) does this agent support"

    capability_specification:
      read_permissions: "What on-chain and off-chain data can the agent access"
      write_permissions: "What state changes can the agent propose or execute"
      governance_layer: "Initial autonomy level (recommended: Layer 1 for new agents)"
      escalation_rules: "When and how the agent escalates to humans"

    integration_specification:
      mcp_dependencies: "Required MCP servers (KOI, Ledger, Tx Builder)"
      plugin_requirements: "ElizaOS plugins needed"
      data_dependencies: "External data sources required"

    resource_requirements:
      service_bond: "REGEN amount to be bonded"
      operational_budget: "Estimated monthly cost from Agent Infrastructure Fund"
      infrastructure: "Compute, storage, API access requirements"

    security_review:
      required: true
      scope:
        - permission_audit: "Verify agent cannot exceed declared permissions"
        - data_access_audit: "Verify agent only accesses declared data sources"
        - failure_mode_analysis: "Enumerate failure modes and their economic impact"
        - adversarial_testing: "Red team exercises against agent decision-making"
      reviewer: "Independent security reviewer (not proposer or agent developer)"

  testing_requirements:
    testnet_deployment:
      duration: "Minimum 30 days on testnet"
      success_criteria:
        - zero_unauthorized_actions
        - accuracy_threshold: ">= 0.85 on test scenarios"
        - escalation_compliance: "All low-confidence decisions properly escalated"
    shadow_mode:
      duration: "Minimum 30 days in production shadow mode (compute but do not execute)"
      success_criteria:
        - shadow_accuracy: ">= 0.80 agreement with human decisions"
        - false_positive_rate: "<= 5%"
        - uptime: ">= 99%"
    community_review:
      - publish_shadow_mode_results
      - 14_day_feedback_period
      - address_community_concerns

  gradual_autonomy_ramp:
    week_1_4: "Layer 1 only — monitoring and reporting, zero autonomous actions"
    week_5_8: "Layer 1 with advisory outputs — agent publishes recommendations but does not act"
    week_9_12: "Layer 2 trial — agent may act on low-risk decisions with 24h override window"
    week_13_plus: "Full Layer 2 (if week 9-12 performance meets criteria) — standard override windows"
```

### Reducing Agent Autonomy

```yaml
autonomy_reduction:
  emergency_override:
    trigger:
      - agent_produces_harmful_output
      - security_vulnerability_discovered
      - agent_confidence_consistently_below_threshold
      - community_reports_concerning_behavior
    authority: >
      Core team (Stage 1), any authority validator (Stage 2), or governance
      multisig (Stage 3) can invoke emergency override.
    action:
      - Immediately halt agent autonomous operations
      - Revert to next lower governance layer (e.g., Layer 1 to monitoring-only)
      - Escalate to Layer 3 human governance for review
      - Publish incident report within 48 hours
      - Agent cannot resume higher autonomy until root cause addressed
    governance_requirement: None (emergency action; ratified post-hoc within 7 days)

  permanent_downgrade:
    governance_layer: Layer 3
    trigger:
      - repeated_override_invocations (>3 in 90 days)
      - community_loss_of_trust (sentiment < 0.50)
      - structural_change_in_domain (agent's capabilities no longer fit)
    process:
      - Governance proposal documenting rationale
      - Community discussion (14 days)
      - Layer 3 vote
    action:
      - Agent's maximum autonomy level is permanently capped at the lower layer
      - Can only be reversed via a new graduation proposal meeting full criteria
```

## Continuous Improvement Framework

The agentic tokenomics system is not a static deployment. This framework ensures ongoing refinement through structured review cycles, feedback loops, validation processes, and community sentiment monitoring.

### Quarterly Review Cycle

```yaml
quarterly_review:
  frequency: Every 3 months (aligned with calendar quarters)
  participants:
    required: [tokenomics_wg, core_team, authority_validators]
    invited: [community_members, agent_maintainers, external_reviewers]

  review_areas:
    1_mechanism_health:
      - M012_supply_dynamics: "Mint/burn balance, distance from equilibrium, cap headroom"
      - M013_fee_performance: "Revenue by type, pool balances, distribution adequacy"
      - M014_validator_health: "Set composition, uptime, compensation sufficiency"
      - M015_reward_distribution: "Activity score distribution, Gini coefficient, stability tier utilization"

    2_agent_performance:
      - accuracy_metrics: "Per-agent decision accuracy vs. human outcomes"
      - escalation_patterns: "Override rates, escalation frequency, false positives"
      - uptime_and_reliability: "Availability, response latency, error rates"
      - community_trust: "Sentiment tracking, override frequency trends"

    3_governance_effectiveness:
      - proposal_throughput: "Proposals submitted, passed, rejected, expired"
      - participation_rates: "Voter turnout, proposer diversity"
      - parameter_change_outcomes: "Post-change monitoring results"
      - open_question_progress: "OQ resolution velocity"

    4_economic_model_validation:
      - token_velocity: "Transaction volume and frequency trends"
      - fee_revenue_sustainability: "Revenue growth vs. infrastructure costs"
      - supply_health: "Circulating supply trajectory vs. modeled expectation"
      - competitive_positioning: "Fee comparison with peer networks"

  outputs:
    - quarterly_health_report: "Published to forum and governance dashboard"
    - parameter_adjustment_recommendations: "Based on review findings"
    - open_question_status_update: "Updated tracker (this document)"
    - roadmap_revision: "If needed, update implementation timeline"

  escalation: >
    If any review area scores below acceptable thresholds (defined per metric),
    an emergency review is convened within 7 days with recommended corrective actions.
```

### Agent Accuracy Feedback Loops

```yaml
agent_feedback_loops:
  real_time_feedback:
    - human_override_logging: >
        Every human override of an agent decision is logged with rationale.
        Agent training data is updated to reflect the correction.
    - community_flag_mechanism: >
        Community members can flag agent outputs as incorrect or concerning.
        Flags above threshold trigger agent maintainer review.
    - accuracy_dashboard: >
        Public dashboard showing per-agent accuracy metrics, updated daily.

  periodic_calibration:
    frequency: Monthly
    process:
      - Aggregate all overrides and flags from past month
      - Identify systematic patterns (e.g., agent consistently overestimates quality scores)
      - Adjust thresholds, weights, or knowledge bases as needed
      - Publish calibration report
    responsible: Agent maintainer team (Stage 1-2), community agent committee (Stage 3)

  cross_agent_learning:
    description: >
      Insights from one agent's performance inform other agents' calibration.
      For example, if AGENT-003's price monitoring reveals systematic bias,
      AGENT-001's quality scoring (which uses price data) is also recalibrated.
    coordination: Quarterly review cycle (see above)

  accuracy_targets:
    agent_001_registry: ">= 0.85 agreement with expert reviewer panel"
    agent_002_governance: ">= 0.80 recommendation adoption rate"
    agent_003_market: ">= 0.90 price anomaly detection accuracy"
    agent_004_validator: ">= 0.95 performance monitoring accuracy"
```

### Economic Model Validation

```yaml
economic_model_validation:
  simulation_framework:
    tool: "cadCAD or equivalent agent-based modeling framework"
    frequency: Semi-annually or on major parameter change proposals
    scope:
      - Supply dynamics (M012) under various fee revenue scenarios
      - Fee routing (M013) distribution adequacy at different volume levels
      - Reward sustainability (M015) stability tier vs. activity pool balance
      - Token velocity and price impact modeling

  empirical_validation:
    frequency: Quarterly (integrated with review cycle)
    process:
      - Compare actual metric values to model predictions from prior quarter
      - Calculate model accuracy scores
      - Identify divergences and root causes
      - Update model parameters to reflect observed reality
    metrics_compared:
      - predicted_vs_actual_fee_revenue
      - predicted_vs_actual_supply_trajectory
      - predicted_vs_actual_stability_tier_demand
      - predicted_vs_actual_governance_participation

  stress_testing:
    frequency: Annually
    scenarios:
      - fee_revenue_collapse: "90% drop in marketplace volume for 6 months"
      - mass_unstaking: "50% of stability tier exits in 30 days"
      - validator_exodus: "Set drops to minimum viable size"
      - token_price_shock: "80% price decline over 3 months"
    output: Resilience report with recommended contingency parameters

  external_review:
    frequency: Annually
    scope: "Independent economic review by qualified party (e.g., Blockscience, Gauntlet)"
    deliverable: Public report with recommendations
```
