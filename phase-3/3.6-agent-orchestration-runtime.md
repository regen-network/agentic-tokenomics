# Phase 3.6: Agent Orchestration Runtime Specification

## 1. Overview

Phase 2.4 defined four agent personas (AGENT-001 Registry Reviewer, AGENT-002 Governance Analyst, AGENT-003 Market Monitor, AGENT-004 Validator Monitor) and their individual capabilities, but left the inter-agent runtime wiring underspecified. This document fills that gap by defining the complete runtime architecture for multi-agent coordination on ElizaOS.

The runtime handles:

- **Inter-agent messaging** via Redis Streams with typed envelopes
- **Shared memory** via PostgreSQL (durable state) and Redis (ephemeral cache)
- **Failure isolation** so one agent's downtime does not cascade
- **Rate limiting** for LLM tokens, MCP requests, and on-chain transactions
- **Supervisory control** for lifecycle management, health monitoring, and graceful shutdown
- **Workflow coordination** for multi-agent sagas with compensating actions

## 2. Inter-Agent Messaging Protocol

### 2.1 Message Envelope Format

Every inter-agent message conforms to a single envelope schema. The envelope is serialized as JSON and written to a Redis Stream entry.

```typescript
/**
 * Canonical message envelope for all inter-agent communication.
 * Serialized as JSON in Redis Stream entries.
 */
interface AgentMessageEnvelope {
  /** Unique message identifier (UUIDv7 for time-ordering) */
  message_id: string;

  /** Agent ID of the sender (e.g., "AGENT-001") */
  from: AgentId;

  /** Agent ID of the recipient, or "BROADCAST" for fan-out */
  to: AgentId | "BROADCAST";

  /** Discriminator for message handling */
  type: MessageType;

  /** Links request-response pairs and saga steps */
  correlation_id: string;

  /** Type-specific data */
  payload: Record<string, unknown>;

  /** Message urgency: P1 (critical) through P4 (background) */
  priority: "P1" | "P2" | "P3" | "P4";

  /** Time-to-live in seconds; consumer MUST discard expired messages */
  ttl: number;

  /** ISO-8601 timestamp of message creation */
  timestamp: string;

  /** Optional reference to the parent saga or workflow */
  saga_id?: string;
}

type AgentId = "AGENT-001" | "AGENT-002" | "AGENT-003" | "AGENT-004" | "SUPERVISOR";

type MessageType =
  | "REQUEST"      // Expects a correlated RESPONSE
  | "RESPONSE"     // Reply to a REQUEST with the same correlation_id
  | "EVENT"        // Fire-and-forget notification
  | "DELEGATION"   // Transfer a sub-task to another agent
  | "BROADCAST";   // Fan-out to all agents via the broadcast stream
```

### 2.2 Message Types

| Type | Semantics | Delivery | Response Expected |
|------|-----------|----------|-------------------|
| `REQUEST` | Ask another agent to perform an action or return data | Point-to-point | Yes (`RESPONSE` with same `correlation_id`) |
| `RESPONSE` | Reply to a prior `REQUEST` | Point-to-point | No |
| `EVENT` | Notify other agents of something that happened | Point-to-point or broadcast | No |
| `DELEGATION` | Transfer ownership of a sub-task to another agent | Point-to-point | Yes (final `RESPONSE` when sub-task completes) |
| `BROADCAST` | Fan-out notification to all agents | Written to `stream:broadcast` | No |

### 2.3 Typed Payload Interfaces

```typescript
/** REQUEST: Ask another agent to perform work */
interface RequestPayload {
  action: string;             // e.g., "review_credit_class", "get_market_context"
  params: Record<string, unknown>;
  timeout_ms: number;         // How long the requester will wait
  idempotency_key?: string;   // Prevents duplicate processing
}

/** RESPONSE: Reply with result or error */
interface ResponsePayload {
  status: "success" | "error" | "timeout" | "rejected";
  data?: Record<string, unknown>;
  error?: {
    code: string;
    message: string;
    retryable: boolean;
  };
  /** If data was served from cache, indicates staleness */
  cache_info?: {
    from_cache: boolean;
    cached_at: string;        // ISO-8601
    stale: boolean;           // true if past normal TTL
  };
}

/** EVENT: Notification about something that occurred */
interface EventPayload {
  event_name: string;         // e.g., "price_anomaly_detected", "proposal_submitted"
  subject_type: string;       // e.g., "credit_class", "proposal", "validator"
  subject_id: string;
  data: Record<string, unknown>;
  severity: "info" | "warning" | "critical";
}

/** DELEGATION: Sub-task handoff */
interface DelegationPayload {
  workflow_id: string;
  step_name: string;
  context: Record<string, unknown>;
  deadline: string;           // ISO-8601 deadline for the delegated task
}

/** BROADCAST: System-wide notification */
interface BroadcastPayload {
  topic: string;              // e.g., "system.health", "market.anomaly", "governance.proposal"
  data: Record<string, unknown>;
  source_event_id?: string;   // Original event that triggered the broadcast
}
```

### 2.4 Redis Streams as Message Bus

Each agent owns an inbox stream. A shared broadcast stream handles fan-out.

| Stream Key | Purpose | Consumers |
|------------|---------|-----------|
| `stream:agent-001` | AGENT-001 inbox | AGENT-001 consumer group |
| `stream:agent-002` | AGENT-002 inbox | AGENT-002 consumer group |
| `stream:agent-003` | AGENT-003 inbox | AGENT-003 consumer group |
| `stream:agent-004` | AGENT-004 inbox | AGENT-004 consumer group |
| `stream:broadcast` | Fan-out events | All agents (one consumer group per agent) |
| `stream:supervisor` | Supervisor commands | AgentSupervisor consumer group |

**Stream configuration:**

```typescript
interface StreamConfig {
  /** Maximum stream length before trimming oldest entries */
  maxlen: number;       // 100_000 entries per agent stream, 500_000 for broadcast
  /** Consumer group name — one per agent per stream */
  group: string;        // e.g., "agent-001-consumers"
  /** Block timeout for XREADGROUP (ms) */
  block_ms: number;     // 5000 (5 seconds)
  /** Number of entries to read per poll */
  count: number;        // 10
}
```

**Producer flow:**
1. Sender serializes `AgentMessageEnvelope` to JSON.
2. Sender writes to the target's inbox stream via `XADD stream:agent-{N} MAXLEN ~ 100000 * envelope {json}`.
3. For `BROADCAST` type, sender writes to `stream:broadcast` instead.

**Consumer flow:**
1. Agent calls `XREADGROUP GROUP {group} {consumer} COUNT 10 BLOCK 5000 STREAMS stream:agent-{N} stream:broadcast > ` to read new messages from both its inbox and the broadcast stream.
2. Agent deserializes the envelope, checks `ttl` (discards expired), routes by `type`.
3. After processing, agent acknowledges via `XACK`.
4. On processing failure, the message remains in the Pending Entry List (PEL) and is re-delivered after the visibility timeout.

### 2.5 Request-Response Pattern

```typescript
/**
 * Send a REQUEST and await the correlated RESPONSE.
 */
async function requestResponse(
  from: AgentId,
  to: AgentId,
  action: string,
  params: Record<string, unknown>,
  timeout_ms: number = 30_000
): Promise<ResponsePayload> {
  const correlation_id = crypto.randomUUID();

  const envelope: AgentMessageEnvelope = {
    message_id: generateUUIDv7(),
    from,
    to,
    type: "REQUEST",
    correlation_id,
    payload: {
      action,
      params,
      timeout_ms,
    } satisfies RequestPayload,
    priority: "P2",
    ttl: Math.ceil(timeout_ms / 1000) + 10, // TTL = timeout + 10s buffer
    timestamp: new Date().toISOString(),
  };

  // Write to target's inbox stream
  await redis.xadd(`stream:${to.toLowerCase()}`, "MAXLEN", "~", "100000", "*",
    "envelope", JSON.stringify(envelope));

  // Await response on a temporary Redis key, or poll own inbox filtered by correlation_id
  const response = await waitForCorrelation(from, correlation_id, timeout_ms);

  if (!response) {
    return {
      status: "timeout",
      error: { code: "TIMEOUT", message: `No response from ${to} within ${timeout_ms}ms`, retryable: true },
    };
  }

  return response.payload as ResponsePayload;
}

/**
 * Wait for a RESPONSE message with the given correlation_id.
 * Uses a Redis pub/sub channel for low-latency notification.
 */
async function waitForCorrelation(
  agent: AgentId,
  correlation_id: string,
  timeout_ms: number
): Promise<AgentMessageEnvelope | null> {
  return new Promise((resolve) => {
    const channel = `correlation:${agent}:${correlation_id}`;
    const timer = setTimeout(() => {
      subscriber.unsubscribe(channel);
      resolve(null);
    }, timeout_ms);

    subscriber.subscribe(channel, (message) => {
      clearTimeout(timer);
      subscriber.unsubscribe(channel);
      resolve(JSON.parse(message));
    });
  });
}
```

### 2.6 Publish-Subscribe for Cross-Cutting Events

Cross-cutting events allow one agent's detection to trigger actions in others. Examples:

| Source Event | Source Agent | Subscribers | Triggered Action |
|---|---|---|---|
| `price_anomaly_detected` | AGENT-003 | AGENT-001 | Review associated credit classes for manipulation |
| `proposal_submitted` | AGENT-002 | AGENT-001, AGENT-003, AGENT-004 | Assess proposal impact on respective domains |
| `validator_slashed` | AGENT-004 | AGENT-002 | Generate governance incident report |
| `credit_class_approved` | AGENT-001 | AGENT-003 | Begin market monitoring for new class |
| `economic_reboot_param_change` | AGENT-002 | AGENT-003, AGENT-004 | Update M012-M015 parameter caches |
| `system.agent_down` | SUPERVISOR | All agents | Adjust cross-agent request behavior (use cache) |
| `system.agent_recovered` | SUPERVISOR | All agents | Resume normal cross-agent requests |

### 3.3 Shared Context Cache in Redis

Frequently accessed cross-agent state is cached in Redis to avoid repeated MCP queries. Each cache entry has a defined TTL and a staleness threshold.

```typescript
/** Redis cache key conventions and TTLs */
const SHARED_CACHE_CONFIG = {
  /** Current governance state: active proposals, voting periods */
  "cache:governance:active_proposals": {
    ttl_seconds: 300,         // 5 minutes
    stale_after_seconds: 600, // 10 minutes (still usable with staleness flag)
    refreshed_by: "AGENT-002",
    consumers: ["AGENT-001", "AGENT-003", "AGENT-004"],
  },

  /** Current market state: prices, volumes, order book summary */
  "cache:market:summary": {
    ttl_seconds: 60,          // 1 minute
    stale_after_seconds: 300, // 5 minutes
    refreshed_by: "AGENT-003",
    consumers: ["AGENT-001", "AGENT-002"],
  },

  /** Current validator set state: performance, delegation totals */
  "cache:validator:status": {
    ttl_seconds: 120,         // 2 minutes
    stale_after_seconds: 600, // 10 minutes
    refreshed_by: "AGENT-004",
    consumers: ["AGENT-002", "AGENT-003"],
  },

  /** Economic reboot parameters: M012-M015 current values */
  "cache:econ_reboot:params": {
    ttl_seconds: 600,         // 10 minutes
    stale_after_seconds: 1800, // 30 minutes
    refreshed_by: "AGENT-002",
    consumers: ["AGENT-003", "AGENT-004"],
  },
} as const;

/**
 * Cache entry wrapper that includes metadata for staleness detection.
 */
interface CacheEntry<T> {
  data: T;
  cached_at: string;          // ISO-8601
  refreshed_by: AgentId;
  version: number;
}

/**
 * Read from shared cache with staleness awareness.
 */
async function getFromCache<T>(key: string): Promise<{ data: T; stale: boolean } | null> {
  const raw = await redis.get(key);
  if (!raw) return null;

  const entry: CacheEntry<T> = JSON.parse(raw);
  const config = SHARED_CACHE_CONFIG[key as keyof typeof SHARED_CACHE_CONFIG];
  const age = (Date.now() - new Date(entry.cached_at).getTime()) / 1000;

  return {
    data: entry.data,
    stale: age > (config?.stale_after_seconds ?? 300),
  };
}
```

### 3.4 State Consistency: Optimistic Locking

All shared mutable tables use a `version` column for optimistic concurrency control. This prevents lost updates when multiple agents modify the same row (e.g., updating saga state).

```sql
-- Example: update saga_state with optimistic lock
UPDATE saga_state
SET
  current_step = 'governance_proposal',
  steps_completed = steps_completed || '["market_context"]'::jsonb,
  context = context || '{"market_data": {"price": 12.50}}'::jsonb,
  version = version + 1,
  updated_at = NOW()
WHERE saga_id = $1
  AND version = $2;  -- Fails if another agent updated first

-- Application MUST check rowcount = 1; if 0, reload and retry
```

```typescript
/**
 * Retry wrapper for optimistic lock conflicts.
 */
async function withOptimisticRetry<T>(
  fn: () => Promise<{ rowCount: number; result: T }>,
  maxRetries: number = 3,
  baseDelay: number = 100
): Promise<T> {
  for (let attempt = 0; attempt < maxRetries; attempt++) {
    const { rowCount, result } = await fn();
    if (rowCount > 0) return result;
    // Exponential backoff with jitter
    const delay = baseDelay * Math.pow(2, attempt) + Math.random() * 50;
    await sleep(delay);
  }
  throw new OptimisticLockError("Failed to acquire optimistic lock after max retries");
}
```

### 3.5 Memory Lifecycle

| Phase | Trigger | Action | Storage |
|-------|---------|--------|---------|
| **Creation** | Agent stores a memory during workflow | Insert into `agent_memories` | PostgreSQL |
| **Active use** | Agent retrieves memories for context | Query by `agent_id`, `memory_type`, similarity search | PostgreSQL |
| **Expiry** | `expires_at` timestamp passed | Cleanup job deletes expired rows | PostgreSQL |
| **Archival** | Memory older than 30 days with `importance_score >= 0.7` | Serialize to KOI object, store IRI in metadata, delete from PostgreSQL | KOI (permanent) |
| **Purge** | Memory older than 30 days with `importance_score < 0.7` | Delete from PostgreSQL | None |

**Cleanup schedule:**

## 4. Failure Handling

### 4.1 Agent Health Check Protocol

Each agent emits a heartbeat to Redis every 30 seconds. The AgentSupervisor monitors these heartbeats and declares an agent down after 90 seconds of silence (3 missed heartbeats).

```typescript
// Agent-side: emit heartbeat
async function emitHeartbeat(agentId: AgentId): Promise<void> {
  const key = `heartbeat:${agentId}`;
  const payload = JSON.stringify({
    agent_id: agentId,
    status: "healthy",
    timestamp: new Date().toISOString(),
    active_workflows: getActiveWorkflowCount(),
    memory_usage_mb: process.memoryUsage().heapUsed / 1024 / 1024,
    pending_messages: getPendingMessageCount(),
  });

  // Set with 90s TTL — if not refreshed within 90s, key auto-expires
  await redis.set(key, payload, "EX", 90);
}

// Supervisor-side: check agent health
```

### 4.3 Graceful Degradation

When an agent is unavailable, the remaining agents continue operating. Cross-agent requests to the down agent use the following fallback strategy:

```typescript
/**
 * Cross-agent request with degradation support.
 * If target agent is down, return cached data with a staleness flag.
 */
async function resilientRequest(
  from: AgentId,
  to: AgentId,
  action: string,
  params: Record<string, unknown>,
  cacheKey?: string
): Promise<ResponsePayload> {
  // Step 1: Check if target agent is healthy
  const health = await checkAgentHealth(to);

  if (health.status === "down") {
    // Step 2: Try to serve from cache
    if (cacheKey) {
      const cached = await getFromCache(cacheKey);
      if (cached) {
        return {
          status: "success",
          data: cached.data as Record<string, unknown>,
          cache_info: {
            from_cache: true,
            cached_at: new Date().toISOString(), // Will be overwritten with actual cache time
            stale: cached.stale,
          },
        };
      }
    }

    // Step 3: No cache available — return graceful error
    return {
      status: "error",
      error: {
        code: "AGENT_UNAVAILABLE",
        message: `${to} is currently down. No cached data available.`,
        retryable: true,
      },
    };
  }

  // Step 4: Agent is healthy — normal request-response
  return requestResponse(from, to, action, params);
}
```

### 4.4 Recovery Protocol

When a previously-down agent recovers:

1. **Reconnect**: Agent re-establishes Redis and PostgreSQL connections.
2. **Resume heartbeat**: Agent begins emitting heartbeats again; Supervisor detects recovery within 30s.
3. **Process queued messages**: Agent reads its inbox stream from the last acknowledged position, processing all queued messages in priority order.
4. **Re-sync state**: Agent queries PostgreSQL for any `saga_state` entries where it is a participant and `status = 'running'`; resumes its role in active sagas.
5. **Broadcast recovery**: Supervisor sends `system.agent_recovered` broadcast so other agents resume normal cross-agent requests.
6. **Cache refresh**: Agent refreshes any shared cache keys it owns (see Section 3.3).

```typescript
async function recoverAgent(agentId: AgentId): Promise<void> {
  // 1. Reconnect (handled by connection pool auto-reconnect)
  await ensureConnections(agentId);

  // 2. Start heartbeat
  startHeartbeatLoop(agentId, 30_000);

  // 3. Process queued messages (from last ACK position)
  const pending = await redis.xreadgroup(
    "GROUP", `${agentId.toLowerCase()}-consumers`, `${agentId.toLowerCase()}-worker`,
    "COUNT", 100, "STREAMS",
    `stream:${agentId.toLowerCase()}`, `stream:broadcast`,
    "0", "0"  // "0" reads all pending (unacknowledged) messages
  );
  for (const message of pending) {
    await processMessage(agentId, message);
  }

  // 4. Re-sync active sagas
  const activeSagas = await db.query(
    `SELECT * FROM saga_state WHERE status IN ('running', 'compensating')
     AND (context->>'participants')::jsonb ? $1`,
    [agentId]
  );
  for (const saga of activeSagas.rows) {
    await resumeSagaParticipation(agentId, saga);
  }

  // 5. Broadcast recovery
  await broadcastEvent("SUPERVISOR", {
    event_name: "system.agent_recovered",
    subject_type: "agent",
    subject_id: agentId,
    data: { recovered_at: new Date().toISOString() },
    severity: "info",
  });

  // 6. Refresh owned caches
  await refreshOwnedCaches(agentId);
}
```

### 5.2 MCP Rate Limits

Rate limits from Phase 2.4 are enforced with a **token bucket algorithm** implemented in Redis.

| MCP Service | Rate Limit | Bucket Capacity | Refill Rate |
|-------------|-----------|-----------------|-------------|
| Ledger MCP | 50 req/min | 50 tokens | ~0.833 tokens/sec |
| KOI MCP | 100 req/min | 100 tokens | ~1.667 tokens/sec |
| Econ Reboot Monitor | 30 req/min | 30 tokens | 0.5 tokens/sec |

```typescript
/**
 * Token bucket rate limiter using Redis for distributed state.
 * Uses a Lua script for atomic check-and-decrement.
 */
const TOKEN_BUCKET_LUA = `
  local key = KEYS[1]
  local capacity = tonumber(ARGV[1])
  local refill_rate = tonumber(ARGV[2])  -- tokens per second
  local now = tonumber(ARGV[3])          -- current timestamp in ms
  local requested = tonumber(ARGV[4])    -- tokens requested (usually 1)

  local bucket = redis.call('HMGET', key, 'tokens', 'last_refill')
  local tokens = tonumber(bucket[1]) or capacity
  local last_refill = tonumber(bucket[2]) or now

  -- Refill tokens based on elapsed time
  local elapsed = (now - last_refill) / 1000
  tokens = math.min(capacity, tokens + elapsed * refill_rate)

  if tokens >= requested then
    tokens = tokens - requested
    redis.call('HMSET', key, 'tokens', tokens, 'last_refill', now)
    redis.call('EXPIRE', key, 120)  -- Auto-cleanup
    return 1  -- Allowed
  else
    redis.call('HMSET', key, 'tokens', tokens, 'last_refill', now)
    redis.call('EXPIRE', key, 120)
    return 0  -- Denied
  end
`;

async function checkMCPRateLimit(service: string, agentId: AgentId): Promise<boolean> {
  const config = MCP_RATE_LIMITS[service];
  const key = `ratelimit:mcp:${service}:${agentId}`;
  const result = await redis.eval(
    TOKEN_BUCKET_LUA, 1, key,
    config.capacity, config.refill_rate, Date.now(), 1
  );
  return result === 1;
}
```

### 5.5 Backpressure

When an agent is rate-limited, incoming work is queued rather than dropped.

```typescript
interface BackpressurePolicy {
  /** Maximum queue depth before rejecting new requests */
  max_queue_depth: number;    // 1000

  /** Priority ordering: P1 items processed first */
  ordering: "priority" | "fifo";

  /** When queue is full, reject with retryable error */
  overflow_action: "reject" | "drop_lowest_priority";
}

/**
 * Enqueue a rate-limited request for later processing.
 * Stored in a Redis sorted set with priority as score.
 */
async function enqueueWithBackpressure(
  agentId: AgentId,
  message: AgentMessageEnvelope
): Promise<void> {
  const queueKey = `backpressure:${agentId}`;
  const depth = await redis.zcard(queueKey);

  if (depth >= 1000) {
    throw new BackpressureOverflowError(`Queue full for ${agentId}`);
  }

  // Score: priority (P1=1, P2=2, P3=3, P4=4) * 1e12 + timestamp for FIFO within priority
  const priorityScore = { P1: 1, P2: 2, P3: 3, P4: 4 }[message.priority];
  const score = priorityScore * 1e12 + Date.now();

  await redis.zadd(queueKey, score, JSON.stringify(message));
}
```

### 6.4 Graceful Shutdown Sequence

When the supervisor receives a `SIGTERM` (e.g., from Kubernetes during pod termination), it executes a controlled shutdown:

```
1. SIGTERM received
   │
2. ├── Stop accepting new messages (remove agents from consumer groups)
   │   └── Broadcast: { event_name: "system.shutting_down" }
   │
3. ├── Drain in-flight messages (wait up to 30s for current processing to finish)
   │
4. ├── Complete in-flight workflows
   │   ├── Short workflows (< 10s estimated): wait for completion
   │   └── Long workflows: checkpoint state to saga_state table, mark as "suspended"
   │
5. ├── Checkpoint all agent state
   │   ├── Flush pending memories to PostgreSQL
   │   ├── Record last-processed stream positions
   │   └── Write shutdown marker to Redis: shutdown:{agent_id} = timestamp
   │
6. ├── Close connections
   │   ├── Redis disconnect
   │   ├── PostgreSQL pool drain
   │   └── MCP client disconnect
   │
7. └── Exit with code 0
```

**Timeout:** If steps 2-5 take longer than `shutdown_timeout_ms` (60s), the supervisor force-kills remaining agents and exits with code 1. Kubernetes allows a configurable `terminationGracePeriodSeconds` (set to 90s to give 30s buffer).

## 7. Workflow Coordination

### 7.1 Multi-Agent Workflows

Many operational scenarios require coordination across multiple agents. Each multi-agent workflow is modeled as a **saga** — a sequence of steps, each owned by a specific agent, with compensating actions for rollback.

### 7.2 Example: Credit Class Application

A new credit class application involves three agents in sequence:

```
Step 1: AGENT-001 — Technical Review
  ├── Review methodology, MRV protocol, additionality
  ├── Query Ledger MCP for similar existing classes
  ├── Produce: ReviewReport { score, recommendation, concerns[] }
  └── Compensate: Mark review as "withdrawn"

Step 2: AGENT-003 — Market Context
  ├── Assess market demand for proposed credit type
  ├── Check existing supply and pricing
  ├── Produce: MarketContext { demand_score, price_range, competition }
  └── Compensate: No-op (read-only step)

Step 3: AGENT-002 — Governance Proposal
  ├── Draft governance proposal combining review + market data
  ├── Submit proposal via Ledger MCP (if review score >= 0.7)
  ├── Produce: ProposalSubmission { proposal_id, tx_hash }
  └── Compensate: Submit withdrawal transaction for the proposal
```

### 7.4 Saga Executor

```typescript
/**
 * Execute a saga, tracking state in PostgreSQL.
 * On step failure, run compensating actions in reverse order.
 */
async function executeSaga(
  definition: SagaDefinition,
  initialContext: Record<string, unknown>
): Promise<SagaResult> {
  // 1. Create saga record
  const sagaId = generateUUIDv7();
  await db.query(
    `INSERT INTO saga_state (saga_id, saga_type, status, current_step, steps_pending, context, initiator_agent)
     VALUES ($1, $2, 'running', $3, $4, $5, $6)`,
    [sagaId, definition.saga_type, definition.steps[0].name,
     JSON.stringify(definition.steps.map(s => s.name)),
     JSON.stringify(initialContext),
     "SUPERVISOR"]
  );

  const completedSteps: SagaStep[] = [];
  let context = { ...initialContext };

  // 2. Execute steps in sequence
  for (const step of definition.steps) {
    try {
      // Update saga state
      await updateSagaStep(sagaId, step.name, "running");

      // Map input parameters from context
      const params: Record<string, unknown> = {};
      for (const [paramKey, contextPath] of Object.entries(step.input_mapping)) {
        params[paramKey] = getNestedValue(context, contextPath);
      }

      // Send DELEGATION message to the responsible agent
      const response = await requestResponse(
        "SUPERVISOR", step.agent, step.action, params, step.timeout_ms
      );

      if (response.status !== "success") {
        throw new SagaStepError(step.name, response.error?.message ?? "Unknown error");
      }

      // Store step result in context
      context[step.output_key] = response.data;
      completedSteps.push(step);

      await updateSagaStep(sagaId, step.name, "completed", context);

    } catch (error) {
      // 3. Step failed — run compensating actions in reverse
      await updateSagaStatus(sagaId, "compensating");

      for (const completed of completedSteps.reverse()) {
        if (completed.compensate) {
          const compensateParams: Record<string, unknown> = {};
          for (const [key, path] of Object.entries(completed.compensate.params_from_context)) {
            compensateParams[key] = getNestedValue(context, path);
          }
          try {
            await requestResponse(
              "SUPERVISOR", completed.agent,
              completed.compensate.action, compensateParams, 30_000
            );
          } catch (compError) {
            // Log compensation failure — requires manual intervention
            await logCompensationFailure(sagaId, completed.name, compError);
          }
        }
      }

      await updateSagaStatus(sagaId, "failed", { error: String(error) });
      return { saga_id: sagaId, status: "failed", error: String(error) };
    }
  }

  // 4. All steps completed
  await updateSagaStatus(sagaId, "completed");
  return { saga_id: sagaId, status: "completed", context };
}

interface SagaResult {
  saga_id: string;
  status: "completed" | "failed";
  context?: Record<string, unknown>;
  error?: string;
}
```

### 7.5 Workflow State Machine

Each saga follows a state machine tracked in the `saga_state` PostgreSQL table:

```
                    ┌──────────┐
                    │ created  │
                    └────┬─────┘
                         │
                    ┌────▼─────┐
               ┌───►│ running  │◄────────┐
               │    └────┬─────┘         │
               │         │               │
               │    step completes   step starts
               │         │               │
               │    ┌────▼─────┐         │
               │    │ step N   ├─────────┘
               │    └────┬─────┘
               │         │
               │    ┌────▼──────────┐
               │    │ all steps     │
               │    │ complete?     │
               │    └──┬────────┬───┘
               │       │yes     │no (failure)
               │  ┌────▼────┐ ┌─▼───────────┐
               │  │completed│ │compensating  │
               │  └─────────┘ └──────┬───────┘
               │                     │
               │              ┌──────▼───────┐
               │              │   failed     │
               │              └──────────────┘
               │
               └──── (recovery: resume from last checkpoint)
```

## 8. Summary

| Aspect | Design Choice | Key Parameters |
|--------|--------------|----------------|
| **Messaging** | Redis Streams with typed JSON envelopes | 1 inbox per agent + 1 broadcast stream; 5 message types |
| **State storage** | PostgreSQL (durable) + Redis (cache) | Optimistic locking with version fields; per-cache TTLs |
| **Agent memory** | `agent_memories` scoped by `agent_id` | 50K max per agent; archive to KOI after 30 days if importance >= 0.7 |
| **Heartbeat** | Redis key with 90s TTL, refreshed every 30s | Supervisor checks every 10s |
| **AGENT-001 down** | Queue reviews; no SLA breach for 24h | P3 alert after 1h, P2 after 12h |
| **AGENT-002 down** | Governance alerts delayed; manual fallback | P3 after 30min, P2 after 6h |
| **AGENT-003 down** | Market monitoring paused | P1 alert after 15min |
| **AGENT-004 down** | Validator monitoring paused | P1 alert after 15min |
| **Graceful degradation** | Cached data with staleness flag | Remaining agents continue independently |
| **Circuit breaker** | Per-MCP-service; 3 failures in 60s opens circuit | Retry after 30s; 2 successes to close |
| **LLM tokens** | 1M/day total; per-agent budgets | AGENT-001: 400K, AGENT-002: 250K, AGENT-003: 200K, AGENT-004: 150K |
| **MCP rate limits** | Token bucket in Redis | Ledger: 50/min, KOI: 100/min, Econ Reboot: 30/min |
| **Transaction limits** | PostgreSQL counters | 10 proposals/day per agent; 50 tx/hour per agent |
| **Backpressure** | Priority sorted set in Redis | Max 1000 queued items per agent |
| **Supervisor** | Single process managing all 4 agents | Exponential backoff restarts; 5 max in 5 min window |
| **Graceful shutdown** | Drain → complete → checkpoint → stop | 60s timeout; K8s grace period 90s |
| **Deployment** | Kubernetes Deployment (replicas: 1, Recreate strategy) | Liveness + readiness probes on :8080 |
| **Multi-agent workflows** | Saga pattern with compensating actions | State machine in `saga_state` table |
| **Workflow example** | Credit class application: AGENT-001 → AGENT-003 → AGENT-002 | Review → Market context → Governance proposal |
