# Phase 4.3: Community Onboarding

## Overview

This document provides a comprehensive onboarding guide for community members participating in the Regen Network Economic Reboot. It covers the transition from inflationary Proof of Stake to revenue-based Proof of Authority, the new fee and reward structures, and the autonomous agent layer that assists with governance, marketplace, and registry operations.

The Economic Reboot replaces four core mechanisms of the Regen Network:

| What | Before | After | Mechanism |
|------|--------|-------|-----------|
| **Token Supply** | Inflationary (open-ended minting) | Hard cap at ~221M REGEN with algorithmic mint/burn | M012 |
| **Transaction Fees** | Flat gas (~0.01 REGEN per tx) | Value-based % fees on credit transactions | M013 |
| **Validator Set** | Open PoS (75 validators, capital-weighted) | Curated PoA (15-21 authority validators, contribution-weighted) | M014 |
| **Staking Rewards** | Passive inflation yield | Activity-based rewards from real fee revenue | M015 |

These changes are deployed in sequence. See the [implementation timeline](#implementation-timeline) for current status.

## 2. Authority Validator Application Process (M014)

Under Proof of Authority, the open validator set is replaced by a curated group of 15-21 authority validators selected for their demonstrated contribution to Regen Network's ecological mission. Validators are compensated from real fee revenue (via M013), not from inflationary token emission.

### 2.1 Validator Categories and Eligibility

Authority validators must belong to one of three categories. The active set maintains a minimum of 5 validators from each category to ensure diverse representation.

#### Category 1: Infrastructure Builders

| Requirement | Detail |
|-------------|--------|
| Active development | Demonstrable work on verification systems, dMRV tools, or registry infrastructure |
| Code contributions | Contributions to `regen-ledger`, ecosystem tooling, or related open-source repositories |
| Operational history | Minimum 6 months of active development or infrastructure operation |
| Examples | RND engineering team, KOI developers, dMRV tool builders |

#### Category 2: Trusted ReFi Partners

| Requirement | Detail |
|-------------|--------|
| Organization status | Established ReFi organization with publicly stated mission alignment |
| Ecosystem participation | Active in credit origination, marketplace activity, or governance on Regen Network |
| Infrastructure capability | Operational infrastructure meeting minimum uptime requirements (99.5%) |
| Examples | ReFiDAO, Toucan, Kolektivo, regional partners |

#### Category 3: Ecological Data Stewards

| Requirement | Detail |
|-------------|--------|
| Data attestation | Organizations attesting to ecological data quality |
| Credit class involvement | Active participation in credit class development or verification processes |
| Domain expertise | Demonstrated expertise in ecology, land management, or environmental science |
| Examples | Verification bodies, research institutions, land steward cooperatives |

### 2.2 Application Submission

Applications are submitted on-chain using `MsgApplyValidator`. The message includes the following fields:

```yaml
# Authority Validator Application
message: MsgApplyValidator
fields:
  applicant:
    type: string
    description: "Your regen1... address (the signing account)"

  operator_address:
    type: string
    description: "Tendermint consensus public key for block production (operator address)"

  category:
    type: string
    description: "One of: INFRASTRUCTURE_BUILDER, REFI_PARTNER, DATA_STEWARD"

  organization_name:
    type: string
    description: "Legal or recognized name of the applying organization"

  evidence_iri:
    type: string
    description: "IRI pointing to supporting evidence package (see 2.3)"

  contact_info:
    type: string
    description: "Primary contact for validator coordination (email or secure channel)"

  description:
    type: string
    description: "Summary of qualifications and mission alignment"

  website:
    type: string
    description: "Organization website URL"
```

**CLI example:**

```bash
regen tx authority apply-validator \
  --category INFRASTRUCTURE_BUILDER \
  --organization-name "Example dMRV Lab" \
  --evidence-iri "regen:13toVfvC2YxEEhYBcFKth3GYtWQpRsZGKiDn2EbBcVXhq1ThBRag5.rdf" \
  --operator-address '{"@type":"/cosmos.crypto.ed25519.PubKey","key":"..."}' \
  --description "Active contributor to dMRV tooling since 2025..." \
  --website "https://example-dmrv.org" \
  --contact-info "validators@example-dmrv.org" \
  --from my-key \
  --chain-id regen-1 \
  --fees 200uregen
```

### 2.3 Evidence Requirements

The `evidence_iri` field must point to a document (anchored on-chain via the `x/data` module) containing verifiable evidence of your qualifications. Accepted evidence varies by category:

| Category | Required Evidence | Recommended Additions |
|----------|-------------------|----------------------|
| Infrastructure Builder | Links to code repositories with commit history; deployed tool URLs; documentation of systems built | Testimonials from ecosystem users; grant completion reports |
| ReFi Partner | Organization charter or mission statement; on-chain transaction history showing marketplace or governance activity; partner references | Media coverage; conference participation records |
| Data Steward | Credentials or institutional affiliation; published research or verification reports; credit class co-development records | Peer-reviewed publications; ISO/ICROA certifications |

**Evidence format**: A JSON-LD document anchored via `MsgAnchor` in the `x/data` module. The IRI returned from anchoring is used as `evidence_iri`. The document should follow the KOI evidence schema and include direct links or hashes of all supporting materials.

### 2.6 Term Structure

| Parameter | Value |
|-----------|-------|
| Term length | 12 months from activation date |
| Compensation review | Quarterly |
| Re-application | Streamlined process for incumbents with good performance records |
| Early exit | Allowed with 30-day notice; forfeits current quarter performance bonus |
| Term expiration | Status transitions to TERM_EXPIRED; must re-apply or gracefully exit |

**For incumbents re-applying**: Validators with no probation events and consistent performance above thresholds enter a streamlined re-evaluation. AGENT-004 compiles a 12-month performance summary, and the governance vote typically proceeds on a shortened timeline.

### 2.7 Performance Expectations

All authority validators must meet the following ongoing requirements:

| Metric | Target | Measurement | Consequence of Failure |
|--------|--------|-------------|----------------------|
| Uptime | 99.5% minimum | Blocks signed / blocks expected (rolling 30-day) | Probation trigger |
| Governance participation | Vote on 80%+ of proposals | Votes cast / proposals during term | Probation trigger |
| Ecosystem contribution | Active contribution per category criteria | Quarterly AGENT-004 assessment | Advisory warning; repeat = probation |
| Software upgrades | Upgrade within required window | Cosmovisor binary switch verification | Probation trigger if upgrade missed |
| Communication | Responsive to coordination channels | Response within 24h to critical alerts | Advisory warning |

**Compensation model**: Validators receive equal base compensation from the Validator Fund (sourced from M013 fee routing). A 10% performance bonus pool is distributed proportionally based on a composite performance score:

```yaml
performance_score:
  uptime_score:              weight: 0.4   # blocks signed / blocks expected
  governance_participation:  weight: 0.3   # votes cast / total proposals
  ecosystem_contribution:    weight: 0.3   # measured by AGENT-004
```

### 2.8 Probation Process

```yaml
probation:
  triggers:
    - "Uptime drops below 99.5% over a rolling 30-day window"
    - "Missed 3+ consecutive governance votes"
    - "AGENT-004 flags performance below threshold"
    - "Governance motion raised by another validator or community member"

  process:
    step_1: "AGENT-004 generates performance report and publishes to KOI"
    step_2: "Validator governance issues formal warning"
    step_3: "Probation period begins (30 days)"
    step_4: "Compensation reduced during probation (base only, no bonus)"

  restoration:
    criteria: "Performance restored above all thresholds for 30 consecutive days"
    verification: "AGENT-004 confirms restoration"
    action: "Full compensation restored; probation flag cleared"

  removal:
    criteria: "Probation period elapses AND performance NOT restored"
    process: "Validator governance confirms removal vote"
    action: "Removed from active set; status set to REMOVED"
    note: "Removed validators may re-apply after a 6-month cooling period"
```

## 3. Stability Tier Enrollment Guide (M015)

The Stability Tier is an optional commitment mechanism within the Contribution-Weighted Rewards system (M015). It provides long-term REGEN holders with a predictable, fixed annual return without requiring active participation in credit markets or governance.

### 3.1 What Is the Stability Tier?

The Stability Tier allows you to lock REGEN tokens for a defined period (6-24 months) and receive a fixed 6% annual return, paid from the Community Pool. This return is funded by real fee revenue flowing through M013 -- not by inflationary token emission.

Think of it as a commitment to network stability: your locked tokens support the stability multiplier in the M012 supply algorithm, which promotes healthier supply dynamics for the entire network.

| Feature | Detail |
|---------|--------|
| Annual return | 6% (paid from Community Pool) |
| Lock period | 6 months minimum, 24 months maximum |
| Minimum commitment | 100 REGEN |
| Early exit | Allowed, with 50% penalty on accrued rewards |
| Cap | Stability Tier allocation limited to 30% of Community Pool inflow per period |
| Reward source | Real fee revenue (M013 community pool share), not inflation |

### 3.2 Eligibility

To enroll in the Stability Tier, you need:

- A Regen Network wallet address with a minimum balance of 100 REGEN
- The tokens you commit must be unbonded (not currently staked or in the unbonding period)
- No maximum per-address limit, but total Stability Tier allocation is capped at 30% of Community Pool inflow

### 3.3 How to Enroll

#### Via CLI

```bash
regen tx rewards commit-stability \
  --amount 1000000000uregen \
  --lock-period 12 \
  --from my-wallet \
  --chain-id regen-1 \
  --fees 5000uregen
```

| Parameter | Description |
|-----------|-------------|
| `--amount` | Number of REGEN tokens to lock (in uregen; 1 REGEN = 1,000,000 uregen). Example: 1000 REGEN = 1000000000uregen |
| `--lock-period` | Lock duration in months (6, 12, 18, or 24) |
| `--from` | Your wallet key name or address |

This submits a `MsgCommitStability` transaction. You will be prompted to confirm the transaction details before signing.

#### Via Wallet UI (Keplr / Regen Dashboard)

1. Navigate to the **Rewards** tab on the Regen Dashboard
2. Select **Stability Tier** from the reward options
3. Enter the amount of REGEN you want to commit (minimum 100 REGEN)
4. Choose your lock period (6, 12, 18, or 24 months)
5. Review the projected return (displayed as total REGEN at maturity)
6. Confirm the transaction in your wallet (Keplr popup)

#### On-Chain State Transition

The commitment creates a `StabilityCommitment` record:

```
UNCOMMITTED --> COMMITTED
  trigger: MsgCommitStability(amount, lock_period)
  guard: amount >= 100 REGEN, lock_period >= 6 months
  action: tokens locked, accrual begins
```

### 3.4 Reward Accrual and Distribution

Rewards accrue continuously and are calculated per distribution period (epoch):

```
reward_per_period = committed_amount * 0.06 / periods_per_year

Example:
  committed_amount = 10,000 REGEN
  periods_per_year = 52 (weekly epochs)
  reward_per_period = 10,000 * 0.06 / 52 = ~11.54 REGEN per week
  total_12_month_return = 600 REGEN
```

Rewards accrue to your commitment record and are paid out at maturity (or at early exit with penalty). They are not auto-compounded -- the accrued amount is calculated on the original commitment.

**Distribution priority**: Each epoch, the Stability Tier allocation is calculated first (capped at 30% of Community Pool inflow). Accrued rewards for all committed holders are calculated proportionally from this allocation. The remaining 70%+ flows to the activity-based reward pool (Section 4).

### 3.5 Early Exit Penalty

You may exit the Stability Tier before your lock period expires, but a penalty applies:

```yaml
early_exit:
  trigger: "MsgExitStability submitted by commitment holder"
  penalty: "50% of all accrued rewards forfeited"
  principal: "100% of original committed tokens returned"
  timing: "Processed in the next epoch after submission"

  example:
    committed: 10,000 REGEN
    lock_period: 12 months
    exit_at: 6 months
    accrued_rewards: 300 REGEN (6 months of 6% annual)
    penalty: 150 REGEN (50% of accrued)
    received: 10,000 REGEN (principal) + 150 REGEN (remaining rewards)
    forfeited: 150 REGEN (returned to Community Pool)
```

Your original tokens are always returned in full. The penalty applies only to accrued rewards.

### 3.6 Maturity and Re-commitment

```yaml
maturity:
  trigger: "Lock period expires"
  state_transition: "COMMITTED --> MATURED"
  action: "Tokens + full accrued rewards unlocked and available"
  auto_renew: "No -- you must explicitly re-commit if desired"
  re_commitment: "Submit a new MsgCommitStability to start a new term"
  gap: "No penalty for time between maturity and re-commitment"
```

At maturity, your tokens and rewards are automatically unlocked. They remain in your account and are available for any use -- re-commitment, governance, marketplace activity, or transfer.

### 3.7 Risk Factors

The Stability Tier is designed for predictability, but participants should understand these risks:

| Risk | Description | Mitigation |
|------|-------------|------------|
| **Revenue dependency** | The 6% return is funded by Community Pool inflow from M013 fees. If marketplace activity drops significantly, the pool may not generate sufficient revenue. | The 30% cap on stability allocation prevents the tier from consuming more than its share. If demand exceeds the cap, new commitments are queued. |
| **Opportunity cost** | Locked tokens cannot be used for activity-based rewards (which may yield more during high-activity periods) or governance voting. | Choose a lock period that matches your investment horizon. Shorter periods (6 months) allow more frequent re-evaluation. |
| **Cap overflow** | If total stability commitments demand more than 30% of Community Pool inflow, new commitments are waitlisted. | Commitments are filled in submission order. Monitor the stability tier utilization on the dashboard before committing. |
| **Smart contract risk** | As with all on-chain systems, the reward module carries implementation risk during initial deployment. | The M015 module goes through a 3-month tracking-only period before distributions begin, allowing for calibration and bug detection. |

### 3.8 Comparison: Stability Tier vs Activity-Based Rewards

| Factor | Stability Tier | Activity-Based Rewards |
|--------|---------------|----------------------|
| **Return type** | Fixed 6% annual | Variable, proportional to your activity score |
| **Effort required** | None after initial commitment | Active participation (buying, retiring, governing) |
| **Token lockup** | Yes (6-24 months) | No lockup; tokens used freely |
| **Predictability** | High -- fixed rate, known maturity date | Low -- depends on your activity relative to all participants |
| **Upside potential** | Capped at 6% | Uncapped; heavy participants in high-activity periods may earn significantly more |
| **Downside risk** | Early exit penalty (50% of accrued rewards) | No penalty, but low activity = low rewards |
| **Governance participation** | Locked tokens cannot vote | Governance voting itself earns reward credit |
| **Pool share** | Up to 30% of Community Pool inflow | At least 70% of Community Pool inflow |
| **Best for** | Long-term holders seeking predictable returns | Active participants in the credit marketplace and governance |

## 4. Activity-Based Rewards Guide (M015)

If you do not enroll in the Stability Tier (or want to earn additional rewards through participation), the Activity-Based Rewards track distributes the remaining Community Pool inflow (at least 70%) to participants based on their on-chain ecological and governance contributions.

### 4.1 What Activities Earn Rewards?

Rewards are calculated using a weighted activity score. Every qualifying on-chain action you perform during a distribution period contributes to your score.

| Activity | Weight | What Counts | Who Benefits |
|----------|--------|-------------|-------------|
| **Credit Purchase** | 30% | Buying ecological credits on the Regen Marketplace | Credit buyers, corporate sustainability teams |
| **Credit Retirement** | 30% | Retiring credits (permanent removal, representing real ecological impact) | Retirees, impact claimants |
| **Platform Facilitation** | 20% | Facilitating transactions as a broker, registry, or marketplace operator | Platform operators, dApp builders |
| **Governance** | 20% | Voting on proposals (10%) and submitting proposals (10%, weighted by outcome) | All REGEN holders, community leaders |

### 4.2 How Composite Scores Are Calculated

Each participant's activity score for a period is computed as follows:

```
activity_score[you] =
    credit_purchase_value   * 0.30
  + credit_retirement_value * 0.30
  + facilitation_value      * 0.20
  + governance_votes_cast   * 0.10
  + proposal_score          * 0.10
```

Your reward for the period is your proportional share of the activity pool:

```
your_reward = activity_pool * (activity_score[you] / total_activity_scores)

where:
  activity_pool = community_pool_inflow - stability_tier_allocation
  community_pool_inflow = community_share * total_fees_collected (from M013)
```

**Proposal scoring details:**

| Proposal Outcome | Score Weight |
|-----------------|-------------|
| Passed (reached quorum and approved) | Full weight (0.10) |
| Failed (reached quorum but rejected) | Half weight (0.05) |
| Did not reach quorum | Zero weight (0.00) |

This prevents spam proposals from inflating activity scores, while still rewarding good-faith proposals that the community engages with.

### 4.3 Claiming Rewards (MsgClaimRewards)

Activity rewards are distributed at the end of each epoch (distribution period). You must claim your rewards explicitly.

#### Via CLI

```bash
regen tx rewards claim-rewards \
  --from my-wallet \
  --chain-id regen-1 \
  --fees 5000uregen
```

#### Via Dashboard

1. Navigate to the **Rewards** tab on the Regen Dashboard
2. View your accrued activity rewards for the current and past periods
3. Click **Claim Rewards**
4. Confirm the transaction in your wallet

Unclaimed rewards remain available indefinitely -- there is no expiration. However, rewards are calculated per period and do not compound. Claiming promptly allows you to redeploy tokens into further activity.

### 4.4 Anti-Gaming Rules

The activity reward system includes several safeguards to prevent manipulation:

| Rule | Description | Rationale |
|------|-------------|-----------|
| **Fee extraction** | Every credit transaction incurs M013 fees (1-3%). Circular wash trading costs more in fees than it earns in rewards. | Sybil resistance through economic friction |
| **Minimum transaction value** | Transactions below the minimum fee floor (1 REGEN equivalent) do not generate activity score credit. | Prevents dust-transaction spam |
| **Proposal quorum requirement** | Proposals that fail to reach quorum earn zero governance score. The existing `x/gov` deposit mechanism provides additional friction. | Prevents governance spam |
| **On-chain only** | Activity scores are derived exclusively from on-chain transactions. No self-reported activity is accepted. | Prevents falsified activity claims |
| **No double-counting** | Each transaction is counted once. Marketplace trades credit the buyer (demand signal); sellers earn credit via issuance and facilitation scores. | Prevents score inflation from single transactions |
| **Single-address attribution** | Activity scores are per-address. Operating multiple addresses does not increase total score because the underlying transaction value is the same. | Fees paid scale with value regardless of address count |

### 4.5 Reward Examples

The following examples illustrate expected rewards under different participation levels. These assume a Community Pool inflow of 50,000 REGEN per epoch (weekly) with a total participant base generating 10,000,000 in aggregate activity score points.

**Example 1: Active credit buyer**

```
Alice purchases $50,000 in carbon credits and retires $20,000 worth during the epoch.

  credit_purchase_value    = 50,000 * 0.30 = 15,000
  credit_retirement_value  = 20,000 * 0.30 =  6,000
  facilitation_value       =      0 * 0.20 =      0
  governance_votes_cast    =      0 * 0.10 =      0
  proposal_score           =      0 * 0.10 =      0
  ─────────────────────────────────────────────────
  activity_score[Alice]    = 21,000

  activity_pool (70% of community pool) = 50,000 * 0.70 = 35,000 REGEN
  Alice's share = 35,000 * (21,000 / 10,000,000) = ~73.5 REGEN for this epoch
  Annualized (52 epochs): ~3,822 REGEN
```

**Example 2: Governance-focused participant**

```
Bob votes on 5 proposals and submits 1 proposal that passes during the epoch.

  credit_purchase_value    =     0 * 0.30 =     0
  credit_retirement_value  =     0 * 0.30 =     0
  facilitation_value       =     0 * 0.20 =     0
  governance_votes_cast    =     5 * 0.10 =   0.5
  proposal_score           =     1 * 0.10 =   0.1
  ─────────────────────────────────────────────────
  activity_score[Bob]      = 0.6

  Bob's share = 35,000 * (0.6 / 10,000,000) = ~0.002 REGEN for this epoch

  Note: Governance-only participation generates modest rewards because vote
  counts are small numbers compared to credit transaction values (measured
  in dollars). Governance weight ensures participation is incentivized, but
  the primary reward signal is ecological activity.
```

**Example 3: Platform operator with diversified activity**

```
Carol operates a retirement platform facilitating $200,000 in transactions,
purchases $10,000 in credits herself, and votes on 3 proposals.

  credit_purchase_value    =  10,000 * 0.30 =  3,000
  credit_retirement_value  =       0 * 0.30 =      0
  facilitation_value       = 200,000 * 0.20 = 40,000
  governance_votes_cast    =       3 * 0.10 =    0.3
  proposal_score           =       0 * 0.10 =      0
  ─────────────────────────────────────────────────────
  activity_score[Carol]    = 43,000.3

  Carol's share = 35,000 * (43,000.3 / 10,000,000) = ~150.5 REGEN for this epoch
  Annualized (52 epochs): ~7,826 REGEN
```

These are illustrative scenarios. Actual rewards depend on total fee revenue flowing through M013, the stability tier allocation, and aggregate participation levels in each epoch.

## 5. Agent Interaction Guides

The agentic tokenomics system includes four autonomous AI agents built on ElizaOS that assist with network operations. These agents operate within the [4-layer governance architecture](../phase-1/1.4-governance-architecture.md): they can observe, analyze, and recommend, but cannot make high-stakes decisions without human oversight.

### 5.1 Agent Overview

| Agent | ID | Role | Governance Layer | Can Propose | Can Execute |
|-------|----|------|-----------------|-------------|-------------|
| Registry Reviewer | AGENT-001 | Reviews credit class applications, project registrations, batch issuances | Layer 2 | Yes | No |
| Governance Analyst | AGENT-002 | Analyzes proposals, tracks voting, publishes governance reports | Layer 1 | No | No |
| Market Monitor | AGENT-003 | Detects price anomalies, monitors liquidity, analyzes retirement patterns | Layer 1-2 | Yes (freeze orders) | No |
| Validator Monitor | AGENT-004 | Tracks validator performance, delegation flows, decentralization metrics | Layer 1-3 | Yes (param changes) | No |

## 6. Fee Structure Guide (M013)

M013 replaces the flat gas fee model with value-proportional fees on ecological credit transactions. Non-credit transactions (standard Cosmos SDK messages like token transfers, governance deposits, IBC transfers) continue to use the existing flat gas fee model.

### 6.1 New Fee Schedule

| Transaction Type | Message | Fee Rate | Minimum Fee |
|-----------------|---------|----------|-------------|
| Credit Issuance | `MsgCreateBatch` | 1-3% of credit value | 1 REGEN |
| Credit Transfer | `MsgSend` (ecocredit) | 0.1% of credit value | 1 REGEN |
| Credit Retirement | `MsgRetire` | 0.5% of credit value | 1 REGEN |
| Marketplace Trade | `MsgBuySellOrder` | 1% of trade value | 1 REGEN |
| Non-credit transactions | Various | Flat gas (unchanged) | Standard gas |

**How credit value is determined**: For marketplace trades, value is explicit (sell order price multiplied by quantity). For issuance, transfer, and retirement, value is estimated using the most recent marketplace price for that credit class. If no marketplace price exists, a governance-set reference price is used.

### 6.2 Fee Distribution Across 4 Pools

Every fee collected is split across four pools that fund different aspects of network operations:

| Pool | Share | Purpose | Recipient |
|------|-------|---------|-----------|
| **Burn Pool** | 30% | Permanent supply reduction via M012 | Protocol (tokens destroyed) |
| **Validator Fund** | 40% | Authority validator compensation (M014) | Active authority validators |
| **Community Pool** | 25% | Activity rewards (M015) + governance-directed spending | Participants and grant recipients |
| **Agent Infrastructure** | 5% | AI agent operational costs | Agent operations budget |

```
Example: A $10,000 credit trade at 1% fee rate

  fee_collected = $10,000 * 0.01 = $100 equivalent in REGEN

  burn_pool       = $100 * 0.30 = $30  --> permanently burned
  validator_fund  = $100 * 0.40 = $40  --> distributed to authority validators
  community_pool  = $100 * 0.25 = $25  --> feeds M015 activity rewards
  agent_infra     = $100 * 0.05 = $5   --> funds agent operations
```

### 6.3 Minimum Fee Floor

All credit transactions have a minimum fee of 1 REGEN, regardless of the calculated percentage. This prevents zero-fee transactions on very low-value credits and provides a baseline for spam prevention.

```
effective_fee = max(calculated_percentage_fee, 1 REGEN)
```

### 6.4 Transition Period: Dual-Fee Mode

The transition from flat gas to value-based fees is phased to allow wallets, dApps, and other tooling to adapt:

```yaml
transition_phases:
  phase_1_dual_mode:
    description: "Both flat gas and value-based fees active simultaneously"
    duration: "90 days (or until governance votes to accelerate)"
    behavior: "Credit transactions pay value-based fee; flat gas remains as backup"
    purpose: "Allow UI/tooling updates; gather real fee data"

  phase_2_value_based:
    description: "Full value-based fees for all credit transactions"
    trigger: "90-day transition period expires OR governance.accelerate()"
    guard: "Fee revenue > 0 for 30 consecutive days"
    behavior: "Flat gas disabled for credit transactions; value-based fees only"
    note: "Non-credit transactions always use flat gas"
```

### 6.5 Before vs After Comparison

| Aspect | Previous (Flat Gas) | New (Value-Based, M013) |
|--------|--------------------|-----------------------|
| Fee amount | ~0.01 REGEN per transaction | 0.1% - 3% of transaction value |
| Revenue model | Negligible protocol revenue | Revenue scales with ecosystem value |
| Fee distribution | 100% to validators | Split: burn (30%), validators (40%), community (25%), agents (5%) |
| Validator funding | Inflation-based staking rewards | Fee revenue from real economic activity |
| Community Pool | Funded by inflation tax | Funded by fee revenue (community pool share) |
| Spam prevention | Gas price floor | Minimum fee (1 REGEN) + percentage fee |
| Non-credit transactions | Flat gas | No change (flat gas preserved) |

### 6.6 Impact on Different User Types

| User Type | What Changes | Net Effect |
|-----------|-------------|------------|
| **Credit Issuers** | Pay 1-3% of credit value at issuance (previously ~0.01 REGEN). This is the highest fee rate, reflecting that issuance is the primary value-creation event. | Higher per-transaction cost, but issuers earn facilitation credit in M015 activity rewards. Elimination of inflation removes the hidden dilution cost. |
| **Credit Traders** | Pay 1% on marketplace trades (previously ~0.01 REGEN). | Moderate increase. Buyers earn activity rewards (30% purchase weight) that may partially offset fees for active traders. |
| **Credit Retirees** | Pay 0.5% at retirement (previously ~0.01 REGEN). | Modest increase. Retirement earns 30% activity weight in rewards, the highest alongside purchases, reflecting the ecological significance of permanent credit retirement. |
| **Credit Transferors** | Pay 0.1% on transfers (previously ~0.01 REGEN). | Minimal impact. Transfer fees are kept low to avoid discouraging credit movement within the ecosystem. |
| **Non-credit Users** | No change. Standard Cosmos SDK transactions (REGEN transfers, IBC, governance deposits) continue to use flat gas. | No impact. |
| **Developers / Integrators** | Must update fee estimation logic to account for value-based fees during the transition period. | One-time integration update required. Agent APIs are available for fee calculation. |

## 7. Frequently Asked Questions

### Staking and PoA Transition

**Q1: What happens to my staked REGEN during the PoA transition?**

Your staked REGEN will need to be unbonded. The transition includes a mandatory, clearly communicated unbonding period. During the PoA coexistence phase (Phase 2 of the M014 rollout), both staking and the new authority validator model operate simultaneously. You will have ample time (the standard 21-day unbonding period plus a minimum 90-day advance notice) to undelegate your tokens. Once unbonded, your tokens are fully available for Stability Tier commitment, marketplace activity, governance voting, or any other use. No staked REGEN is lost -- you simply need to actively undelegate.

**Q2: How do I become an authority validator?**

Submit an application on-chain via `MsgApplyValidator`, providing evidence of your qualifications in one of three categories: Infrastructure Builder, ReFi Partner, or Ecological Data Steward. Your application undergoes agent screening (24-48 hours), community discussion (14+ days on the forum), and a governance vote (7-day voting period). See [Section 2](#2-authority-validator-application-process-m014) for the complete process, including evidence requirements and CLI commands. The total timeline from submission to activation is approximately 4-6 weeks.

**Q3: What is the Stability Tier and is it safe?**

The Stability Tier is an optional commitment within M015 that offers a fixed 6% annual return in exchange for locking your REGEN for 6-24 months. It is funded by real fee revenue from the Community Pool (not inflation), which makes it more economically grounded than previous staking yields. The primary risks are: (a) if marketplace activity is low, the 30% Community Pool cap limits how much can be allocated to stability returns; and (b) your tokens are locked for the chosen period (early exit incurs a 50% penalty on accrued rewards). Your principal is always returned in full. See [Section 3.7](#37-risk-factors) for the complete risk assessment.

**Q4: How are activity rewards calculated?**

Your reward is proportional to your activity score relative to all other participants in each epoch. The score is calculated from five weighted activities: credit purchase (30%), credit retirement (30%), platform facilitation (20%), governance voting (10%), and proposal submission (10%). Your reward equals the activity pool (at least 70% of Community Pool inflow) multiplied by your share of total activity scores. See [Section 4.2](#42-how-composite-scores-are-calculated) for the full formula and [Section 4.5](#45-reward-examples) for worked examples with numbers.

**Q5: What do the AI agents do?**

Four agents assist with network operations: AGENT-001 (Registry Reviewer) reviews credit class applications and project registrations; AGENT-002 (Governance Analyst) analyzes proposals and tracks voting; AGENT-003 (Market Monitor) monitors marketplace prices, liquidity, and retirement patterns; AGENT-004 (Validator Monitor) tracks validator performance and network decentralization. They observe, analyze, and recommend -- but cannot make final decisions on significant matters without human approval. All four operate within the 4-layer governance model described in [Section 5](#5-agent-interaction-guides).

**Q6: Can agents make decisions without human approval?**

It depends on the governance layer. At Layer 1 (monitoring and alerts), agents operate fully autonomously because they only observe and report -- no state changes are made. At Layer 2 (routine operations), agents propose actions that auto-execute after a 24-72 hour human override window if nobody objects. At Layer 3 and above (significant and constitutional decisions), agents provide analysis and recommendations, but humans always make the final decision through governance votes. Agents can never execute Layer 3 or Layer 4 actions.

**Q7: What happens if an agent makes a mistake?**

Layer 2 decisions have a mandatory override window (24 or 72 hours depending on significance) during which any community member can request human review. If a mistake is detected after execution, the decision can be reversed through the normal governance process (a new proposal to undo the action). All agent decisions are logged in KOI with full audit trails: the observations the agent considered, its reasoning, its confidence score, and the evidence it relied upon. Agent confidence thresholds are tuned so that low-confidence decisions are automatically escalated to human review. The system is designed to fail toward human oversight, not away from it.

**Q8: How do I interact with agents?**

Agents are available through three channels: (1) **Discord**, where you can mention an agent by name in its designated channel (e.g., `@GovernanceAnalyst summarize proposal #65`); (2) the **Regen Forum**, where AGENT-002 automatically posts governance analysis threads; and (3) the **Regen Dashboard**, where you can view agent decisions, pending escalations, historical records, and submit override requests. See [Section 5](#5-agent-interaction-guides) for detailed commands and example conversations per agent.

**Q9: What changed about marketplace fees?**

Flat gas fees (~0.01 REGEN) have been replaced with value-proportional fees: 1-3% on issuance, 0.1% on transfers, 0.5% on retirements, and 1% on marketplace trades. These fees fund the network sustainably: 30% is burned (reducing supply), 40% compensates validators, 25% funds community rewards, and 5% supports agent infrastructure. Non-credit transactions (token transfers, IBC, governance) still use flat gas fees. See [Section 6](#6-fee-structure-guide-m013) for the complete schedule and distribution breakdown.

**Q10: Is the REGEN supply increasing or decreasing?**

Under M012, the supply is dynamic within a hard cap of approximately 221M REGEN. New tokens are minted at a rate that decelerates as supply approaches the cap (the regrowth model: `M[t] = r * (C - S[t])`). Simultaneously, tokens are burned from M013 fee revenue. When minting and burning reach equilibrium, supply stabilizes. Whether supply increases or decreases in any given period depends on the balance between the regrowth rate and the burn rate. The system is designed to reach equilibrium over time, unlike the previous model where inflation was continuous and open-ended.

**Q11: What is the hard cap and can it change?**

The hard cap is approximately 221M REGEN, matching the current total supply. It can only be changed through a Layer 4 constitutional governance vote requiring a 67% supermajority of the authority validator set. This is the highest governance threshold in the system and is intentionally difficult to modify, providing long-term monetary policy certainty comparable to Bitcoin's fixed supply cap.

**Q12: How is the Community Pool funded now?**

Previously, the Community Pool received a share of inflationary block rewards. Under M013, the Community Pool receives 25% of all value-based fee revenue from ecological credit transactions. This means the pool is funded by real economic activity on the network (credit issuance, trading, retirement) rather than by dilutive inflation. The Community Pool serves two purposes: (a) automatic distribution via M015 activity rewards and stability tier returns, and (b) governance-directed spending via community pool spend proposals (the existing process continues to function as before).

**Q13: Can I participate in governance under PoA?**

Yes. Governance participation actually becomes more rewarding under PoA. The governance model transitions from capital-weighted (1 REGEN = 1 vote) to a dual-track system: authority validators vote with equal weight per validator, and token holders participate with contribution-weighted scores (derived from M015 activity). Additionally, governance participation (voting and proposals) is one of the five weighted activities that earn rewards under M015, directly incentivizing engagement. You do not need to be a validator to participate in governance.

**Q14: What if I disagree with an agent's recommendation?**

You have several options depending on the decision type. For Layer 2 decisions (routine operations), you can trigger a human override during the override window (24-72 hours) via Discord (`@AgentName override decision #123`), the Dashboard (click "Request Review"), or on-chain (`MsgRequestOverride`). This escalates the decision to Layer 3 human governance. For informational outputs (Layer 1), such as proposal analysis or market reports, you can post corrections or alternative analysis in the relevant Discord channel or forum thread. Agent recommendations are advisory -- they never replace human judgment on significant matters. See [Section 5.5](#55-how-to-trigger-a-layer-2-override-human-override-window) for the complete override process.

**Q15: Where can I track the Economic Reboot progress?**

Multiple channels provide real-time and periodic updates:
- **Regen Dashboard**: Real-time mechanism status, fee revenue, supply metrics, and reward distributions
- **Forum**: Monthly Economic Reboot Progress Reports from the Tokenomics WG
- **Discord (#announcements)**: Milestone activations and transition phase notices
- **GitHub** ([regen-network/agentic-tokenomics](https://github.com/regen-network/agentic-tokenomics)): Specification updates, implementation progress, and open questions
- **Newsletter**: Monthly digest covering all changes, metrics, and upcoming events

The [Phase 4.2 Mainnet Migration Plan](./4.2-mainnet-migration.md) provides the detailed deployment sequence and stage gates.

### Additional Questions

**Q16: I am currently a validator. Am I automatically included in the authority set?**

No. All validators must apply through the authority validator application process. However, current validators who meet the category criteria and have strong track records will have a natural advantage during the application review. The transition is designed to be inclusive of existing contributors while establishing a formal quality bar. During the coexistence phase, current validators continue operating normally.

**Q17: Can I participate in both the Stability Tier and Activity Rewards?**

Only with different tokens. Tokens committed to the Stability Tier are locked and cannot be used for marketplace transactions that earn activity scores. You can split your holdings: commit some to the Stability Tier for predictable returns, and use the rest for marketplace activity to earn activity-based rewards. This allows you to balance predictability against upside potential.

**Q18: I am a credit buyer. What changes for me?**

Credit purchases now incur a 1% marketplace fee (previously negligible gas). In exchange, your purchases earn activity rewards from the Community Pool (30% weight on credit purchases). If you are a regular buyer, the rewards may partially or fully offset the increased fees, while also giving you governance weight proportional to your economic contribution.

**Q19: I am a developer building on Regen. How does this affect me?**

If you build tools that facilitate credit transactions (marketplaces, retirement interfaces, broker platforms), your platform earns facilitation credit (20% activity weight) in the reward system. You should update your fee estimation logic to account for M013 value-based fees during the transition period. Agent APIs are available for integration -- see the [Technical Documentation](../phase-3/3.5-technical-docs.md) for endpoint specs.

**Q20: How can I verify that agents are operating correctly?**

All agent decisions are logged in KOI (Regen's knowledge graph) with full audit trails. You can query any agent's decision history through the Dashboard, the Agent API, or by resolving the KOI object IRIs directly. Each decision record includes: the trigger event, all observations the agent gathered, its orientation analysis, the decision made (with confidence score), the actions taken, and the evidence relied upon. AGENT-004 also publishes regular validator performance reports that are publicly accessible.
